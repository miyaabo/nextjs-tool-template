# Implementation Rules

この文書は、AIがコード、ファイル構成、UI、命名、実装粒度を提案するときのルールを定義する。

## 1. 実装の前提
- まず最小構成で動くものを作る
- 1画面で完結する形を優先する
- 将来拡張のための過剰な抽象化をしない
- 共通化は必要になってから行う
- 初手で複雑な状態管理を入れない
- 実装速度を優先する

## 2. ファイル構成ルール

### 小規模ツールの基本構成例
- `index.html` / `style.css` / `script.js`

### React系で必要最小限の構成例
- `src/App.jsx`
- `src/components/`
- `src/utils/`
- `src/styles/`

### ルール
- 1ツールにつき責務を分けすぎない
- `hooks/`, `types/`, `services/` などを無理に増やさない
- 外部通信がないならAPI層は作らない

## 3. 命名規則
- ツールID: kebab-case
- Reactコンポーネント: PascalCase
- 関数名: camelCase
- 定数: UPPER_SNAKE_CASE
- localStorageキー: `tool:<tool-id>:<name>` 形式

避ける名前: `temp`, `data`, `info`, `item`, `value1`, `value2`

## 4. UIルール

### 基本
- 主要操作は1画面内で完結
- 入力欄の意味が見て分かる
- 結果は入力の近くに表示
- コピーボタンを用意する
- リセット操作を用意する
- 使用例を最低1つ置く
- スマホ幅でも最低限使えるようにする

### 推奨
- 入力項目は原則5個以内
- メインボタンは1つ
- ボタン文言は動詞で終える（例: 生成する、整形する）

### 避けるもの
- タブの多用 / 多段フォーム / 過剰なアニメーション

## 5. 勝手に追加してはいけないもの

明示的な指示がない限り追加しないこと:
- ログイン・認証機能
- DB保存
- バックエンドAPI
- 外部AI API依存
- 決済機能
- 多言語対応
- ダークモード切り替え
- 管理画面
- 通知・お気に入り機能

## 6. ライブラリ導入ルール
- 標準機能で足りるならライブラリを入れない
- UIライブラリは原則入れない
- 状態管理ライブラリは原則入れない

## 7. localStorage利用ルール
- 保存が必要なときのみ使う
- 保存するのは軽い設定や直近入力程度に留める
- 保存名は用途が分かる形にする

## 8. 公開前の最低要件
- ツール名で用途が分かる
- 入力欄にラベルがある
- 結果が表示される
- 結果をコピーできる
- リセットできる
- 使用例がある
- スマホ幅で致命的に崩れない
- エラー時に無反応にならない
